Simple Modeling Language

07 Dec 2012

I've wrote about the State of the ORMs and how it is essential to have a well defined model first.

When I was looking for ways to design my models, I was quite surprised that it is such a pain in the ass. There's a huge demand for this and a tremendous amount of work has been done but they all seem unintuitive and overly complicated.

For instance take a look at propel's modeling language

<?xml version="1.0" encoding="UTF-8"?>
<database name="bookstore" defaultIdMethod="native">
  <table name="book" phpName="Book">
    <column name="id" type="integer" required="true" primaryKey="true" autoIncrement="true"/>
    <column name="title" type="varchar" size="255" required="true" />
    <column name="isbn" type="varchar" size="24" required="true" phpName="ISBN"/>
    <column name="publisher_id" type="integer" required="true"/>
    <column name="author_id" type="integer" required="true"/>
    <foreign-key foreignTable="publisher" phpName="Publisher" refPhpName="Book">
      <reference local="publisher_id" foreign="id"/>
    </foreign-key>
    <foreign-key foreignTable="author">
      <reference local="author_id" foreign="id"/>
    </foreign-key>
  </table>
  <table name="author" phpName="Author">
    <column name="id" type="integer" required="true" primaryKey="true" autoIncrement="true"/>
    <column name="first_name" type="varchar" size="128" required="true"/>
    <column name="last_name" type="varchar" size="128" required="true"/>
  </table>
  <table name="publisher" phpName="Publisher">
   <column name="id" type="integer" required="true" primaryKey="true" autoIncrement="true" />
   <column name="name" type="varchar" size="128" required="true" />
  </table>
</database>

And I'm just ranting how it could be defined, just bare with me.

<book>
  <title max="255"/>
  <isbn:int len="13"/>
  <publisher:publisher/>
  <author:author/>
</book>

<author>
  <first_name max="128"/>
  <last_name max="128"/>
  <books:book[]/>
</author>

<publisher>
  <name max="128"/>
  <books:book[]/>
</publisher>

I think any programmer who is familiar with array accessor operator [] would immediately understand what about model means exactly.

Defaults

* Each table has a INT PK AI (Primary Key, Auto Increment)

* Each field is required=true and type=text. I put the max="128" just to represent as much information as the original XML does.

Defining cardinality (relationship association types)

one-to-one

Imagine each book can has only one author and each author can only have one book.

<book>
  ...
  <author:author/>
</book>

<author>
  ...
  <book:book/>
</author>

This should generate either author_id field in the book entity OR book_id in the author entity.

Note that this schema is also applies for one-to-many. If enforcing one is required, then a unique index can be created for author_id or book_id field.

one-to-many

Imagine each author can have many books (naturally).

<book>
  ...
  <author:author/>
</book>

<author>
  ...
  <book:book[]/>
</author>

This will generate author_id in the book entity and that's it.

many-to-many

It is perfectly possible for one book to have many authors, so here's what we do.

<book>
  ...
  <author:author[]/>
</book>

<author>
  ...
  <book:book[]/>
</author>

Then, we should generate a book_author entity with fields book_id, author_id. Then we can insert as many book-author links as we want.

There might be cases which this syntax fail to define which I can't think of right now.

Syntax

We can use verbose element attributes such as

<title type="string" required="true" len="10" index="true"/>

Or we can extract common properties of each element and innovate a shorthand syntax for them.

field[*?][:type[(min,max)]] [|validator_function]

Example Hint Description
<name/> Default text Name field of type text.
<name*/> Index Name field of type text and indexed if possible (if allowed by underlying repository technology)
<name?/> Nullable Name field of type text and accepts empty values.
 <name*?/> Indexable, Nullable Name field of type text which accepts empty values and
 <age:int/> Data types Age field of type int.
 <age:int(18,100)/> Minimum and maximum ranges Age field of type int where accepted values are between 18 and 100.
 <name:text(3,32)/>  Minimum and maximum length Name field which must be at least 3 characters and at most 32 characters.
 <license|uk_license_validator/> Validators License field of type text and will be validated with a function called uk_license_validator whom stub will be created.
<phone*?:text(7,12)|phone_validator> Mixing up stuff Phone field of type text whom length can be between 7 and 12 and will be validated with phone_validator.
